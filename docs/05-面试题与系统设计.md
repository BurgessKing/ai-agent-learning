#### Q11：RAG 和 Fine-tuning 怎么选择？实际项目中你是怎么决策的？

**答案**：

```
决策三问：
1. 知识更新频率高吗？ → 高=RAG，低=Fine-tuning
2. 需要对模型"思维方式"改造吗？ → 是=Fine-tuning，否=RAG  
3. 有足够高质量标注数据吗？ → 有=Fine-tuning，无=RAG

我的实战决策树（面试详细版）：

场景A: 公交运营手册问答
→ 选RAG：手册定期更新，需要引用原文出处，不需要改变模型风格

场景B: 公交客服回复风格统一（礼貌、专业、标准话术）
→ 选Fine-tuning + RAG：微调让模型学会"公交客服的说话方式"，RAG提供准确知识

场景C: 散包报价逻辑（考虑车型、距离、时间、天气的综合定价）
→ 选Agent + 规则引擎：报价涉及精确计算，不适合纯LLM
  用Agent调用定价API + RAG辅助解释定价逻辑

成本对比（7B模型为例）：
- RAG: 每次查询约$0.001（Embedding + LLM调用）
- Fine-tuning: 训练一次$50-200，推理成本不变
- 日活1万查询，RAG花$10/天，Fine-tuning 20天回本
```

#### Q12：Agent 开发中你遇到过哪些坑？怎么解决的？

**答案**（用你的公交项目讲）：

```
坑1: 工具返回数据太大，超出 Context Window
- 场景: 查100路30天客流数据，返回5万条记录
- 解决: 
  ① 工具层先做聚合（返回统计摘要而非原始数据）
  ② 大结果分页 + 让Agent按需查询下一页
  ③ 工具返回时加上 summary 字段（前100字概要）

坑2: Agent "幻觉"编造车辆信息
- 场景: Agent 说"浙A88888在3号站台"但这个车不存在
- 解决: 
  ① System Prompt 加强："所有车辆信息必须来自工具查询，不确定时说'请稍等，我帮您查询'"
  ② 对关键信息做后校验（车牌号正则匹配数据库）
  ③ 在工具返回中添加 is_verified 标记

坑3: 多轮对话中 Agent 丢失上文
- 场景: 用户说"刚才查的那班车呢"，Agent 不知道是哪班
- 解决:
  ① 使用滑动窗口记忆（保留最近5轮完整对话）
  ② 对更早的对话做摘要压缩（用LLM提取关键信息）
  ③ 用 Redis 持久化会话状态（route_id, last_query 等结构化字段）

坑4: 并发下 Agent 状态混乱
- 场景: 两个用户同时申请临时班车，调度了同一辆车
- 解决:
  ① 每用户独立 Agent 实例 + 独立 session
  ② 车辆分配使用分布式锁（Redis SETNX）
  ③ 数据库乐观锁兜底
```

#### Q13：解释 MCP (Model Context Protocol) 是什么？为什么重要？

**答案**：

```
MCP = Anthropic 提出的 Agent-工具标准协议

类比：HTTP 之于 Web，MCP 之于 Agent-工具通信

解决的问题：
  Before MCP: 每个Agent框架用自己的方式定义工具
    LangChain: @tool 装饰器
    CrewAI:    BaseTool 子类
    AutoGen:   Function 注册
    结果: 换个框架就要重写工具代码

  After MCP: 统一的 Client-Server 协议
    Tool Server: 暴露标准化的工具接口
    Agent Client: 通过 MCP 协议调用任意 Server 的工具
    结果: 一次编写，所有 Agent 框架通用

架构:
┌──────────┐   MCP协议    ┌──────────────┐
│ Agent    │ ←──────────→ │ MCP Server   │
│ (Client) │  JSON-RPC    │ (Tool Server)│
└──────────┘              └──────┬───────┘
                                 │
                    ┌────────────┼────────────┐
                    ↓            ↓            ↓
              ┌──────────┐ ┌──────────┐ ┌──────────┐
              │ Database │ │  API     │ │  File    │
              │ Server   │ │  Server  │ │  Server  │
              └──────────┘ └──────────┘ └──────────┘

面试话术:
"MCP 正在成为 Agent 生态的标准协议。我们在架构中已经开始预留 MCP 接口，
让公交业务系统的能力（路线查询、调度分配）通过 MCP Server 暴露，
未来可以无缝接入各种 Agent 框架。"
```
